## 第9章 编码风格

### 9.3 标识符命名

1. 常用缩写惯例

   > `count`写成`cnt`，`block`写成`blk`，`length`写成`len`，`window`写成`win`，`message`写成`msg`，`number`写成`nr`，`temporary`可以写成`temp`，也可以进一步写成`tmp`

2. 变量、函数和类型采用全小写加下划线的方式命名，常量（比如宏定义和枚举常量）采用全大写加下划线的方式命名

3. 全局变量和全局函数的命名一定要详细，不惜多用几个单词多写几个下划线。

4. 局部变量和只在一个源文件中调用的内部函数的命名可以简略一些

### 9.4 函数

1. 函数内部的缩进层次不宜过多，一般以少于4层为宜。
2. 执行函数就是执行一个动作，函数名通常应包含动词，例如`get_current`、`radix_tree_insert`



## 第11章 排序与查找

### 11.1 算法的概念

算法（Algorithm）是将一组输入转化成一组输出的一系列计算步骤，其中每个步骤必须能在有限时间内完成。

**算法必须能解决一类问题**，而非某个特定问题。（通用性）



### 11.2 插入排序

#### 1. 核心意象：摸牌与理牌

想象你正在打扑克牌（比如斗地主）。

- 你的**左手**：是你已经整理好的、从小到大排列的牌（**已排序区**）。
- 你的**右手**（或者桌面上）：是还没抓完的一堆乱牌（**未排序区**）。

**场景：** 现在你的左手有三张牌：`[3, 5, 8]`。这时，你用右手从桌上摸起了一张 `6`。

**问：** 在现实中，为了把 `6` 放进左手，你大脑和手部肌肉会执行哪几个微观动作？

> **思考提示：** 你是一把抓起 `6` 直接和 `8` 交换吗？还是有一个“拨开”的动作？

**答：**

1. **暂存：** 你的右手紧紧捏住这张 `6`（在 C 语言中，这叫 `int key = arr[i]`，把它存入寄存器或局部变量）。
2. **比较与腾位置：** 你的眼睛从右向左扫视左手的牌。看到 `8` 比 `6` 大，于是你的大拇指把 `8` 往右边拨动一格，腾出一个空隙（在 C 语言中，这叫 `arr[j+1] = arr[j]`，内存数据的平移）。
3. **落位：** 继续看 `5`，发现 `5` 比 `6` 小，停止拨动！直接把捏在手里的 `6` 插进刚刚腾出的空位（在 C 语言中，这叫 `arr[j+1] = key`）。

```c
void insertion_sort(int arr[], int n)
{
    for (int i = 1; i < n; i++)
    {
        int key = arr[i];
        int j = i - 1;
        while (j >= 0 && arr[j] < key)
        {
            arr[j + 1] = arr[j];
            j--;
        }
        arr[j + 1] = key;
        for (int k = 0; k < n; k++)
        {
            printf("%d ", arr[k]);
        }
        printf("\n");
    }
}
```



### 11.3 算法的时间复杂度分析

#### 插入排序的时间复杂度分析

##### 场景一：最完美的“顺风局”（最好情况）

假设你手里的牌已经是 `[1, 2, 3, 4, 5]` 了。 

活的量级和 $n$ 成正比。所以 **最好时间复杂度是 $O(n)$**。

> **底层视角：** 这种情况下，CPU 的分支预测器（Branch Predictor）会非常开心，因为它每次都猜对“不用进循环”，流水线几乎不会断。

------

##### 场景二：最倒霉的“逆序局”（最坏情况）

假设牌是完全反过来的 `[5, 4, 3, 2, 1]`。

**问：**

1. 当你拿到 `4` 时，需要往后挪多少个数？（答：1 个，`5` 挪走）
2. 当你拿到 `3` 时，需要往后挪多少个数？（答：2 个，`5, 4` 都要挪）
3. 当你拿到最后那个 `1` 时，需要挪多少个？（答：$n-1$ 个）

**问：** 这是一个等差数列求和。$1 + 2 + 3 + ... + (n-1)$ 等于多少？
 **答：** $\frac{n(n-1)}{2}$，也就是 $\frac{1}{2}n^2 - \frac{1}{2}n$。

**结论：** 在大 $O$ 表示法里，我们只看最高阶项并忽略系数。所以 **最坏时间复杂度是 $O(n^2)$**。

------

##### 场景三：随机的“平均局”（平均情况）

假设牌是乱序的。

**问：** 每次摸起一张新牌，平均来说，它应该插在已排序区域的什么位置？开头？结尾？还是中间？
 **答：** 概率上来说，平均会插在**中间**。
 **问：** 那意味着每次内层循环平均要跑 $\frac{j}{2}$ 次（$j$ 是当前已排序的数量）。
 **问：** 把这些 $\frac{1}{2}$ 累加起来，结果会变成什么？
 **答：** 结果依然是 $\frac{1}{4}n^2$ 左右。

**结论：** 依然是 $n^2$ 级别的，所以 **平均时间复杂度是 $O(n^2)$**。

**在分析算法的时间复杂度时，我们更关心最坏情况而不是最好情况**

几种常见的时间复杂度函数按数量级从小到大的顺序依次是：Θ(lgn)，Θ(sqrt(n))，Θ(n)，Θ(nlgn)，Θ(n2)，Θ(n3)，Θ(2n)，Θ(n!)。

Θ的含义和“等于”类似，而大O的含义和“小于等于”类似。

> **$O$ (Big-O)：** 它是**上界**（Upper Bound）。像是一句承诺：“我的代码跑得再慢，也不会超过这个速度。”（$\leq$）
>
> **$\Omega$ (Big-Omega)：** 它是**下界**（Lower Bound）。像是一句底线：“我的代码跑得再快，起码也得花这么多时间。”（$\geq$）
>
> **$\Theta$ (Big-Theta)：** 当上界和下界重合时，它就是**确界**（Tight Bound）。它宣布：“这个算法的增长量级，**就是**这个样子的。”（$=$）

#### 主定理

适用于递归类函数的时间复杂度分析

如果一个递归函数符合这种模式：
 $T(n) = a \cdot T\left(\frac{n}{b}\right) + O(n^d)$

- $a$：你把大问题分裂成了几个子问题？（归并排序里 $a=2$）
- $b$：每个子问题的规模是原来的几分之一？（归并排序里 $b=2$）
- $d$：合并结果那一小步的复杂度指数是多少？（归并排序里 $O(n^1)$，所以 $d=1$）

主定理直接告诉你怎么看结果：

1. 如果 $a < b^d$，那么 $T(n) = O(n^d)$
2. 如果 $a = b^d$，那么 $T(n) = O(n^d \log n)$
3. 如果 $a > b^d$，那么 $T(n) = O(n^{\log_b a})$



### 11.4 归并排序

归并排序，它采取分而治之（Divide-and-Conquer）的策略，时间复杂度是Θ(nlgn)。归并排序的步骤如下：

1. **分（Divide）**：把数组从中间切开，变成左、右两半。
2. **治（Conquer）**：递归地对这两半分别进行归并排序（直到每份只剩一个数字）。
3. **合（Combine）**：把两个已经排好序的小数组，合并成一个大的有序数组。

```c
// merge two sorted subarrays a[start..mid] and a[mid+1..end]
void merge(int start, int mid, int end)
{
    int n1 = mid - start + 1;
    int n2 = end - mid;
    int left[n1], right[n2];
    for (int i = 0; i < n1; i++)
        left[i] = a[start + i];
    for (int i = 0; i < n2; i++) // 修复 #1: 使用 n2 作为循环条件
        right[i] = a[mid + 1 + i];
    int i = 0, j = 0;
    int k = start;
    while (i < n1 && j < n2)
    {
        if (left[i] < right[j])
            a[k++] = left[i++];
        else
            a[k++] = right[j++]; // 修复 #2: 当 right[j] 更小时，使用 j++
    }

    // copy remaining elements of left[] if any
    while (i < n1)
        a[k++] = left[i++];
    // copy remaining elements of right[] if any
    while (j < n2)
        a[k++] = right[j++];
}

// merge sort
void sort(int start, int end)
{
    if (start < end)
    {
        int mid = (start + end) / 2;
        sort(start, mid);
        sort(mid + 1, end);
        merge(start, mid, end);
    }
}
```

#### 归并排序的时间复杂度分析

我们设 $T(n)$ 为**“对长度为 $n$ 的数组进行归并排序所花费的总时间”**。

```
sort(start, mid);         // 动作 1：排左半边
sort(mid+1, end);         // 动作 2：排右半边
merge(start, mid, end);   // 动作 3：合并
```

**问：** 动作 3 是将两个排好序的数组合并，我们之前分析过，它的时间复杂度是多少？
 **答：** 我们需要把这 $n$ 个元素全扫一遍，所以是 $O(n)$（或者写成 $c \cdot n$，其中 $c$ 是常数）。

**结论：** 我们得到了计算机科学中最著名的一个递推方程：
 $T(n) = 2 \cdot T\left(\frac{n}{2}\right) + O(n)$
 *(当 $n=1$ 时，不需要排序，所以边界条件是 $T(1) = O(1)$)*

怎么解这个方程？最直观的方法是在脑海里画一棵**递归树**。我们要把这棵树上**所有节点花费的时间全部加起来**。

- **第 1 层（树根）：**
   你调用了 `sort(0, n-1)`。这一层本身的合并工作量是 $n$。
   它裂变出了两个孩子：处理 $n/2$ 的子任务。
- **第 2 层：**
   有两个节点，每个节点要处理 $n/2$ 的数据。
   **问：** 这两个节点的合并工作量加起来是多少？
   **答：** $2 \times \frac{n}{2} = n$。
- **第 3 层：**
   有四个节点，每个节点处理 $n/4$ 的数据。
   **问：** 这四个节点的合并工作量加起来是多少？
   **答：** $4 \times \frac{n}{4} = n$。

无论这棵树裂变到哪一层，**这一层所有节点的合并工作量加在一起，永远都是 $n$**

既然每一层的总工作量都是 $n$，我们只需要知道**这棵树一共有多少层**，乘起来就完事了。

**问：** 数组长度从 $n$ 开始，每次除以 2，除多少次会变成 1（到达叶子节点，即递归出口）？
 **答：** $\log_2 n$ 次。所以树的高度大约是 $\log_2 n$ 层。

**最终结算：**
 总时间 $T(n)$ = 每层的工作量 $\times$ 层数
 $T(n) = n \times \log_2 n$
 所以，归并排序的时间复杂度是 **$O(n \log n)$**。

#### 快速排序 

##### 1. 核心意象：选大哥与站队

想象你面对着一排身高参差不齐的新兵。
 归并排序是“无脑对半劈开”，而快速排序的逻辑是**“找个标杆，重新站队”**。

1. **选大哥（Pivot / 基准值）**：你随便从队伍里拉出一个人（比如第一个人）。
2. **站队（Partition / 划分）**：你大喊一声：“比大哥矮的，统统站到左边！比大哥高的，统统站到右边！”
3. **递归（分治）**：现在大哥站在了中间（他这辈子的最终位置已经确定了）。然后你对左边那群人和右边那群人，重复刚才的动作。

------

##### 2. 如何在“原地”完成站队？

**场景：** 数组是 `[6, 1, 2, 7, 9, 3, 4, 5, 10, 8]`。假设我们选第一个数 `6` 作为大哥（Pivot）。

**问：** 如果不给你额外的数组空间，你只有左右两只手（两个指针 `i` 和 `j`），你怎么在原数组里把比 6 小的扔左边，比 6 大的扔右边？

> **思考提示：** 想象你的左手 `i` 在最左端，右手 `j` 在最右端。它们向中间相向而行。

**答：**
 这是经典的 **Hoare 划分法**（或者其变体）：

1. **右手 `j` 先出动**：从右往左找，找一个**比 6 小**的数。找到了 `5`，停住。
2. **左手 `i` 再出动**：从左往右找，找一个**比 6 大**的数。找到了 `7`，停住。
3. **物理交换（Swap）**：现在 `i` 指向 `7`，`j` 指向 `5`。直接把它们俩交换！数组变成了 `[..., 5, ..., 7, ...]`。
4. **继续走**：直到 `i` 和 `j` 碰头相遇。相遇的位置，就是大哥 `6` 该待的真正位置。把 `6` 和相遇位置的数交换。

这整个过程中，**没有任何多余的内存分配**。

```c
int partition(int start, int end)
{
    // 1. 选定 pivot，并在逻辑上在 a[start] 挖出一个“坑”
    // 在底层，pivot 被分配在当前函数的栈帧或直接放入 CPU 寄存器中
    int pivot = a[start]; 
    
    int left = start;
    int right = end;

    // 当 left 和 right 还没有相遇时，继续腾挪
    while (left < right) {
        
        // 2. right 先动：从右向左找【小于】 pivot 的数
        // 注意：必须加上 left < right 的越界保护！
        while (left < right && a[right] >= pivot) {
            right--;
        }
        // 找到了！把右边的数填入左边的“坑”，此时 a[right] 变成了新坑
        a[left] = a[right]; 

        // 3. left 后动：从左向右找【大于】 pivot 的数
        while (left < right && a[left] <= pivot) {
            left++;
        }
        // 找到了！把左边的数填入右边的“坑”，此时 a[left] 又变成了新坑
        a[right] = a[left];
    }

    // 4. left 和 right 相遇了！这里就是 pivot 命中注定的位置
    // 把一直捏在手里（寄存器里）的 pivot 填入最后一个坑
    a[left] = pivot;

    // 返回相遇的位置，这就是 mid
    return left; 
}

void quicksort(int start, int end)
{
    int mid;
    // 只有当区间至少有 2 个元素时才需要排序
    if (start < end) { 
        mid = partition(start, end);
        // pivot 已经就位，接下来只需排它左边和右边的兄弟
        quicksort(start, mid - 1);
        quicksort(mid + 1, end);
    }
}
```

### 11.5 线性查找

在一组随机排列的数中找出第k小的，这个元素称为k-th Order Statistic。能想到的最直观的算法肯定是先把这些数排序然后取第k个，时间复杂度和排序算法相同，可以是Θ(nlgn)。它也有平均情况下时间复杂度是Θ(n)的算法，将上一节习题1的快速排序算法稍加修改就可以解决这个问题：

```c
int partition(int start, int end)
{
    //...
    //同上个程序
}

/* 从start到end之间找出第k小的元素 */
int order_statistic(int start, int end, int k)
{
    //用partition函数把序列分成两半，中间的pivot元素是序列中的第i个;
    int i = partition(start, end);
    if (k == i)
        return a[i];
    else if (k > i)
        //从后半部分找出第k-i小的元素并返回;
        order_statistic(i + 1, end, k - i);
    else
        //从前半部分找出第k小的元素并返回;
        order_statistic(start, i, k);
}
```



### 11.6 折半查找

每次取中间的元素和待查找的元素比较，如果中间的元素比待查找的元素小，就说明“如果待查找的元素存在，一定位于序列的后半部分”，这样可以把搜索范围缩小到后半部分，然后再次使用这种算法迭代。这种“每次将搜索范围缩小一半”的思想称为**折半查找（Binary Search）**。

```c
int binarysearch(int number)
{
	int mid, start = 0, end = LEN - 1;

	while (start <= end) {
		mid = (start + end) / 2;
		if (a[mid] < number)
			start = mid + 1;
		else if (a[mid] > number)
			end = mid - 1;
		else
			return mid;
	}
	return -1;
}
```

#### 契约式编程 (Design by Contract, DbC)

在系统级编程中，函数不再只是代码块，而是调用者（Caller/老板）和被调用者（Callee/打工人）之间订立的**严格契约**。

测试一个函数是否绝对正确，必须覆盖以下三个维度：

1. **Precondition (前置条件) —— 乙方的免责声明**
   - **含义：** 调用函数**前**，Caller 必须保证的状态。
   - **例如：** “我帮你找数字可以，但你传给我的数组**必须是已经排好序的**！”
2. **Postcondition (后置条件) —— 乙方的交付承诺**
   - **含义：** 函数返回**时**，Callee 承诺达到的结果。
   - **例如：** “如果数字存在，我保证返回正确的下标；如果不存在，我保证返回 `-1`。”
3. **Maintenance / Invariant (循环不变量) —— 过程中的底线**
   - **含义：** 在循环（如二分查找不断缩小区间）的过程中，**永远不能被破坏的逻辑真理**。
   - **例如：** “不管区间怎么缩小，**如果目标值存在，它永远不可能跑到 `[start, end]` 这个圈子外面去**。”（一旦跑出去，说明边界更新写错了，比如 `start = mid` 写成了 `start = mid + 2`）。

#### 防御性编程的终极武器：`assert` (断言)

不要用 `printf` 调试底层代码！要用 `assert` 让程序在出错的第一时间“原地爆炸”。

- **头文件：** `#include <assert.h>`
- **用法：** `assert(条件);`
  - 条件为真：无事发生，继续运行。
  - 条件为假：立即调用 `abort()` 终止程序，并在终端精准打印出出错的文件名、行号和失败的条件代码。

**在代码中的体现：**

```
int binarysearch(int number) {
    assert(is_sorted()); // 进门先验货：校验 Precondition
    
    while (start <= end) {
        assert(mustbe(start, end, number)); // 每次缩小圈子前校验：维护 Invariant
        // ... 二分查找逻辑 ...
    }
    
    assert(!contains(number)); // 没找到准备返回前校验：兑现 Postcondition
    return -1;
}
```

#### `NDEBUG` 宏

**致命矛盾：** 二分查找本身极快 $O(\log n)$，但为了测试加进去的 `assert(is_sorted())` 需要遍历数组，是极慢的 $O(n)$。如果上线运行，会导致性能雪崩。

**完美解决方案：**
 在包含头文件之前定义宏 `#define NDEBUG`（或者在编译时加上参数 `gcc -DNDEBUG`）。

- **Debug（开发阶段）：** 默认状态，所有的 `assert` 都生效，严格捕获每一个微小的逻辑 Bug。
- **Release（发布阶段）：** 开启 `NDEBUG` 后，C 语言预处理器会将代码中**所有的 `assert` 替换为空**。
  - 生成的汇编/机器码中连测试代码的影子都没有。
  - 完美恢复 $O(\log n)$ 的极致性能！
  - 这就是系统编程中典型的**“零成本测试/零成本抽象”**。

## 课程概览与shell

### The shell

如果某个路径以 `/` 开头，那么它是一个 *绝对路径*，其他的都是 *相对路径* 。

相对路径是指相对于当前工作目录的路径，当前工作目录可以使用 `pwd` 命令来获取

切换目录需要使用 `cd` 命令。在路径中，`.` 表示的是当前目录，而 `..` 表示上级目录

为了查看指定目录下包含哪些文件，我们使用 `ls` 命令

**在执行程序时使用 `-h` 或 `--help` 标记可以打印帮助信息，以便了解有哪些可用的标记或选项**

也可以使用`man`查看程序，如`man touch`

`touch NEWFILE`可以创建一个内容为空的新文件`NEWFILE`, 若`NEWFILE`已存在, 其内容不会丢失

#### 在程序间创建连接

最简单的重定向是 `< file` 和 `> file`。这两个命令可以将程序的输入输出流分别重定向到文件

```bash
missing:~$ echo hello > hello.txt
missing:~$ cat hello.txt
hello
missing:~$ cat < hello.txt
hello
missing:~$ cat < hello.txt > hello2.txt
missing:~$ cat hello2.txt
hello
```

还可以使用 `>>` 来向一个文件追加内容

`|` 操作符允许我们将一个程序的输出和另外一个程序的输入连接起来

#### 课后习题

用 `ls` 看权限：

```bash
ls -l ./semester
```

最前面的权限位里如果没有 `x`，就说明它不能被直接执行

`chmod` 是用来修改文件权限的：

```bash
chmod +x semester
```



## Shell工具和脚本

### Shell脚本

在 bash 中为变量赋值的语法是 `foo=bar`，访问变量中存储的数值，其语法为 `$foo`。 需要注意的是，`foo = bar` （使用空格隔开）是不能正确工作的（**空格会起到分割参数的作用**）

Bash 中的字符串通过 `'` 和 `"` 分隔符来定义，但是它们的含义并不相同。以 `'` 定义的字符串为原义字符串，其中的变量不会被转义，而 `"` 定义的字符串会将变量值进行替换。

```bash
foo=bar
echo "$foo"
# 打印 bar
echo '$foo'
# 打印 $foo
```

`$0`：脚本是谁

`$1...$9`：脚本参数

`$@`：所有参数

`$#`：参数数量

`$?`：上条命令成功没

`$$`：当前进程号

`!!`：上一整条命令

`$_`：上一条命令最后一个参数



#### shell 的 *通配*（*globbing*）

- 通配符 - 当你想要利用通配符进行匹配时，你可以分别使用 `?` 和 `*` 来匹配一个或任意个字符。例如，对于文件 `foo`, `foo1`, `foo2`, `foo10` 和 `bar`, `rm foo?` 这条命令会删除 `foo1` 和 `foo2` ，而 `rm foo*` 则会删除除了 `bar` 之外的所有文件。
- 花括号 `{}` - 当你有一系列的指令，其中包含一段公共子串时，可以用花括号来自动展开这些命令。这在批量移动或转换文件时非常方便。



### Shell工具

#### 查找文件

`find` 命令会递归地搜索符合条件的文件，例如：

```bash
# 查找所有名称为src的文件夹
find . -name src -type d
# 查找所有文件夹路径中包含test的python文件
find . -path '*/test/*.py' -type f
# 查找前一天修改的所有文件
find . -mtime -1
# 查找所有大小在500k至10M的tar.gz文件
find . -size +500k -size -10M -name '*.tar.gz'
```

#### 查找代码

`grep` 有很多选项，这也使它成为一个非常全能的工具。其中我经常使用的有 `-C` ：获取查找结果的上下文（Context）；`-v` 将对结果进行反选（Invert），也就是输出不匹配的结果。举例来说， `grep -C 5` 会输出匹配结果前后五行。当需要搜索大量文件的时候，使用 `-R` 会递归地进入子目录并搜索所有的文本文件。



## 数据整理

###  一、核心基石

#### 1. `grep` 

**作用**：在文件中过滤、搜索包含特定“关键词”的行。

**核心选项 (Attributes)**：

| 选项     | 含义         | 实战场景                                                   |
| -------- | ------------ | ---------------------------------------------------------- |
| **`-n`** | 显示行号     | 报错日志里找到 "Error" 在哪一行，直接定位。                |
| **`-r`** | 递归搜索     | `grep -r "vaddr_read" .` (在整个 NEMU 项目里找谁调用了它)  |
| **`-i`** | 忽略大小写   | `grep -i "error" log.txt` (Error, error, ERROR 都找出来)   |
| **`-E`** | 启用扩展正则 | `grep -E "Error                                            |
| **`-w`** | 精确匹配单词 | `grep -w "pc" log.txt` (只找 pc，不会匹配到 space)         |
| **`-v`** | 反向匹配     | `grep -v "Warning"` (过滤掉所有烦人的 Warning，只看 Error) |

#### 2. `tail` 

**作用**：专门看文件的“尾巴”，因为 Bug 现场永远在日志的最后。

**核心选项 (Attributes)**：

| 选项            | 含义          | 实战场景                                                 |
| --------------- | ------------- | -------------------------------------------------------- |
| **`-n <行数>`** | 显示最后 N 行 | `tail -n 200 run.log` (NEMU 崩溃了，直接看最后200条指令) |
| **`-f`**        | 实时追踪      | `tail -f build.log` (开个分屏实时监控编译过程，看卡在哪) |

### 3. `wc` 

**作用**：统计文件的行数、单词数、字符数。

**核心选项 (Attributes)**：

| 选项     | 含义                 | 实战场景                                           |
| -------- | -------------------- | -------------------------------------------------- |
| **`-l`** | 只统计行数 (lines)   | `wc -l itrace.log` (看看 CPU 一共执行了多少条指令) |
| **`-w`** | 只统计单词数 (words) |                                                    |
| **`-c`** | 只统计字节数 (bytes) |                                                    |

------

###  二、

#### 1. `awk` (列提取器)

**作用**：按列切割文本，并格式化输出

**核心语法**：`awk '{print $N}'`，其中 `$N` 代表第 N 列。

| 语法                   | 含义             | 实战场景                                                     |
| ---------------------- | ---------------- | ------------------------------------------------------------ |
| **`'{print $1}'`**     | 打印第一列       | `awk '{print $1}' itrace.log > pc.txt` (把指令 trace 里的 PC 地址那一列单独提取出来，方便对比) |
| **`'{print $1, $3}'`** | 打印第一和第三列 |                                                              |
| **`-F <分隔符>`**      | 指定列的分隔符   | `awk -F ':' '{print $2}'` (如果日志是用冒号分隔的)           |

### 2. `sort` & `uniq`

**作用**：`sort` 负责排序，`uniq` 负责去重和计数。两者通常用管道符 `|` 连用。

**核心选项 (Attributes)**：

| 命令       | 选项     | 含义                           |
| ---------- | -------- | ------------------------------ |
| **`sort`** | (无)     | 按字典序排序                   |
|            | **`-n`** | 按数值大小排序                 |
| **`uniq`** | **`-c`** | 去重，并统计每一项重复了多少次 |

------

### 三、自动化与批量处理

#### `sed` (The Stream Editor / 流编辑器)

**作用**：在 Shell 脚本中进行非交互式的查找和替换，适合做批量修改。

**核心语法**：`sed 's/旧内容/新内容/g'`

| 语法                | 含义                                     | 实战场景                                                     |
| ------------------- | ---------------------------------------- | ------------------------------------------------------------ |
| **`'s/old/new/g'`** | s=substitute, g=global，替换行内所有匹配 |                                                              |
| **`-i`**            | 直接修改文件本身 (in-place)              | `sed -i 's/vaddr_read/paddr_read/g' main.c` (把 `main.c` 里的所有 `vaddr_read` 替换成 `paddr_read`) |

## 编辑器（Vim）

### 移动

多数时候你会在正常模式下，使用移动命令在缓存中导航。在 Vim 里面移动也被称为 “名词”， 因为它们指向文字块。

- 基本移动: `hjkl` （左， 下， 上， 右）
- 词： `w` （下一个词）， `b` （词初）， `e` （词尾）
- 行： `0` （行初）， `^` （第一个非空格字符）， `$` （行尾）
- 屏幕： `H` （屏幕首行）， `M` （屏幕中间）， `L` （屏幕底部）
- 翻页： `Ctrl-u` （上翻）， `Ctrl-d` （下翻）
- 文件： `gg` （文件头）， `G` （文件尾）
- 行数： `:{行数}<CR>` 或者 `{行数}G` ({行数}为行数)
- 杂项： `%` （找到配对，比如括号或者 /* */ 之类的注释对）
- 查找： `f{字符}`， `t{字符}`， `F{字符}`， `T{字符}`
  - 查找/到 向前/向后 在本行的{字符}
  - `,` / `;` 用于导航匹配
- 搜索: `/{正则表达式}`, `n` / `N` 用于导航匹配

### 选择

可视化模式:

- 可视化：`v`
- 可视化行： `V`
- 可视化块：`Ctrl+v`

可以用移动命令来选中。

### 编辑

所有你需要用鼠标做的事， 你现在都可以用键盘：采用编辑命令和移动命令的组合来完成。 这就是 Vim 的界面开始看起来像一个程序语言的时候。Vim 的编辑命令也被称为 “动词”， 因为动词可以施动于名词。

- `i` 进入插入模式
  - 但是对于操纵/编辑文本，不单想用退格键完成
- `O` / `o` 在之上/之下插入行
- `d{移动命令}` 删除 {移动命令}
  - 例如，`dw` 删除词, `d$` 删除到行尾, `d0` 删除到行头。
- `c{移动命令}` 改变 {移动命令}
  - 例如，`cw` 改变词
  - 比如 `d{移动命令}` 再 `i`
- `x` 删除字符（等同于 `dl`）
- `s` 替换字符（等同于 `xi`）
- 可视化模式 + 操作
  - 选中文字, `d` 删除 或者 `c` 改变
- `u` 撤销, `<C-r>` 重做
- `y` 复制 / “yank” （其他一些命令比如 `d` 也会复制）
- `p` 粘贴
- 更多值得学习的: 比如 `~` 改变字符的大小写

### 计数

你可以用一个计数来结合“名词”和“动词”，这会执行指定操作若干次。

- `3w` 向后移动三个词
- `5j` 向下移动 5 行
- `7dw` 删除 7 个词

### 修饰语

你可以用修饰语改变“名词”的意义。修饰语有 `i`，表示“内部”或者“在内”，和 `a`， 表示“周围”。

- `ci(` 改变当前括号内的内容
- `ci[` 改变当前方括号内的内容
- `da'` 删除一个单引号字符串， 包括周围的单引号



## 版本控制(Git)

### 一、Git 的世界观：快照 (Snapshot) > 差异 (Diff)

- **传统认知误区**：很多人以为版本控制是记录“每次改了哪几行代码”（增量差异）。
- **Git 的真相**：Git 记录的是每一次提交时，整个项目文件系统的**完整快照**。

### 二、Git 的三大核心对象 (Objects)

在 Git 的眼里，代码世界只有三种基本元素（它们在底层都被视为“对象”）：

1. **Blob（数据对象）**：
   - 本质：就是**文件内容本身**（一组字节数据）。
   - 注意：它只存内容，不存文件名！
2. **Tree（树对象）**：
   - 本质：就是**目录/文件夹**。
   - 作用：它像一个字典，将“文件名”映射到对应的 Blob（文件）或其他 Tree（子目录）。
3. **Commit（提交对象）**：
   - 本质：对某一次快照的**打包和描述**。
   - 包含内容：
     - `snapshot`：指向顶层 Tree 的指针。
     - `parents`：指向它**之前的一个或多个 Commit**（合并操作会有两个父辈）。
     - `author` & `message`：作者和提交说明。

### 三、历史记录的形态：有向无环图 (DAG)

- **不是一条直线**：Git 的历史不是简单的按时间排序的直线，而是一个**有向无环图**。
- **指针朝向**：当前提交永远指向它的**父辈（过去）**，而不是未来。
- **不可变性 (Immutable)**：一个 Commit 一旦生成，就**绝对无法更改**。

### 四、底层寻址机制：SHA-1 哈希

- **哈希寻址**：Git 不通过路径或文件名在底层找东西，而是通过 **SHA-1 哈希值**（一串 40 位的十六进制字符，如 `698281bc...`）来寻找任何 Blob、Tree 或 Commit。
- **牵一发而动全身**：因为哈希是由内容计算得来的，只要文件内容改了一个标点符号，Blob 的哈希就会变 $\rightarrow$ 导致包含它的 Tree 的哈希变 $\rightarrow$ 导致 Commit 的哈希变。**这保证了 Git 历史的绝对安全和不可篡改。**

### 五、引用 (References) 与 HEAD

因为人类记不住 40 位的哈希值，Git 引入了“引用”机制。

- **Reference (引用)**：一个指向特定 Commit 哈希值的**可变指针**（具有人类可读的名字）。
  - *例子*：`master` 或 `main` 本质上就是一个引用，当你提交新代码时，这个指针会自动向前移动，指向最新的 Commit。
- **HEAD (当前位置)**：一个超级特殊的引用。
  - 它代表**“你当前在这个图谱里的哪个位置”**。
  - 它通常指向一个分支（如 `master`），但在“分离头指针（Detached HEAD）”状态下，它可以直接指向某个具体的 Commit 哈希。

### 六、工作流的缓冲带：暂存区 (Staging Area)

- **为什么需要暂存区？** 如果没有它，每次 `git commit` 都会把工作目录里所有的改动打包。但实际开发中，你可能同时修了 Bug A 和 Bug PA 实验模块 B，你想把它们分成两次干净的提交。
- **本质**：它是你**“组装下一次快照的草稿箱”**。通过 `git add`，你挑选特定的改动放入这个草稿箱，最后 `git commit` 时，Git 只看草稿箱里的内容来生成新的 Commit。

## Finite State Machines

### 状态机简介

#### 1. 什么是状态机

状态机 = **会记忆“当前状态”** 的时序电路。

它不是只看当前输入，还要看：

```
当前状态 + 输入 -> 下一状态
当前状态 -> 输出
```

所以状态机本质上就是：

```
组合逻辑 + 状态寄存器 + 反馈
```

------

#### 2. 状态机的三个核心部分

写 Verilog 状态机时，通常分成 3 部分：

##### ① 状态寄存器

在时钟沿把 `next_state` 存进 `state`

```verilog
always @(posedge clk) begin
    if (reset)
        state <= S0;
    else
        state <= next_state;
end
```

------

##### ② 下一状态逻辑

根据 **当前状态** 和 **输入**，决定下一拍跳到哪里

```verilog
always @(*) begin
    case (state)
        S0: begin
            if (in)
                next_state = S1;
            else
                next_state = S0;
        end
        S1: begin
            if (in)
                next_state = S1;
            else
                next_state = S0;
        end
        default: next_state = S0;
    endcase
end
```

------

##### ③ 输出逻辑

根据状态（或者状态+输入）决定输出

```verilog
always @(*) begin
    case (state)
        S0: out = 1'b0;
        S1: out = 1'b1;
        default: out = 1'b0;
    endcase
end
```

------

### 3. 用图理解状态机

一个圆圈 = 一个状态。

例如图里有两个状态：

```
S0: out = 0
S1: out = 1
```

箭头表示：

```
在某个输入条件下，从当前状态跳到另一个状态
```

------

### 4. Moore 状态机 vs Mealy 状态机

#### Moore 型

输出只由 **状态** 决定：

```
out = f(state)
```

特点：

- 输出写在圆圈里
- 更稳定，容易理解

#### Mealy 型

输出由 **状态 + 输入** 决定：

```
out = f(state, input)
```

特点：

- 输出常写在箭头上
- 反应更快，但更容易抖动

#### 判断方法

- **输出写在状态圆里** → Moore
- **输出写在转移箭头上** → Mealy

------

### 5. 状态机最核心的思维

每次做题先问自己三句话：

#### 第一句：我现在在哪个状态？

```
state
```

#### 第二句：根据输入，下一拍去哪？

```
next_state
```

#### 第三句：当前状态输出什么？

```
out
```

### 6. 做题步骤

以后做状态机题，按这个顺序：

#### 第一步：画/看状态图

先搞清楚：

- 有几个状态
- 每个状态输出什么
- 输入条件下怎么跳转

#### 第二步：给状态命名

例如：

```
parameter S0 = 0, S1 = 1;
```

#### 第三步：写状态寄存器

```
always @(posedge clk)
```

#### 第四步：写 next_state

```
always @(*)
```

#### 第五步：写输出逻辑

```
always @(*)
```



## one-hot编码的FSM

#### one-hot 的核心思路

> 每个状态一根线，哪根线为 1 就表示当前在哪个状态。

#### 写 next_state 的方法

> 对于 `next_state[i]`，把所有“会转移到该状态”的条件 OR 起来。
>
> ##### 先看 `next_state[A]`
>
> 哪些情况会转到 A？
>
> 从表里看：
>
> - `A` 在 `in=0` 时去 `A`
> - `C` 在 `in=0` 时去 `A`
>
> 所以：
>
> ```verilog
> next_state[0] = state[0] & ~in | state[2] & ~in;
> ```

#### 写输出的方法

> 哪些状态输出为 1，就把对应状态位 OR 起来。

|            | 二进制编码 | one-hot    |
| ---------- | ---------- | ---------- |
| 状态位数   | 2 bit      | 4 bit      |
| 状态判断   | 需要比较   | 直接一根线 |
| 逻辑复杂度 | 更复杂     | 非常简单   |
| 寄存器数量 | 少         | 多         |

**注意：复位时必须有一位是1**



## localparam和parameter的区别

`parameter` 和 `localparam` 的核心区别在于**“参数是否允许从外部修改”**。

**`parameter` 是给别人看的接口，`localparam` 是把自己锁死在内部的私有配置。**

| 特性         | `parameter`                        | `localparam`                       |
| ------------ | ---------------------------------- | ---------------------------------- |
| **可修改性** | 可以通过 `defparam` 或模块例化修改 | **绝对不可修改**                   |
| **应用场景** | 定义位宽、频率、ID 等配置项        | 定义状态编码、内部常量、固定查找表 |
| **可见性**   | 对模块外部可见                     | 仅在模块内部可见                   |

用 `parameter` 定义状态是为了：

1. **零成本**：不消耗芯片物理资源。
2. **高可读性**：让你看代码时知道 `A` 是什么，而不是对着满屏幕的 `2'b00` 发呆。
3. 相当于logisim里的那个常量标签



## 某状态持续太久会触发某结果

遇到“某状态持续太久会触发某结果”的题，最稳的方法一般是：

- **一个 FSM 管行为**
- **一个计数器管持续时间**
- **一个 flag 管是否超时**

这比把所有东西都揉进 `next_state` 里稳得多。

**如果 FSM 某个行为和“持续时间”有关，优先考虑用“状态机 + 独立计数器 + 超时标志”三件套，而不是直接在 next_state 里硬算拍数。**



## 状态机防漏方法

**1. 划定主线长度（Happy Path）**

- 只看最顺利的情况，画出一条直线的状态链。
- **意义**：这决定了你状态机的“基本盘”，你能明确知道到底需要几个寄存器 bit 来存这些状态。

**2. 建立“二元拷问矩阵”（穷举法）**

- 在草稿纸上画一个表格，强迫自己把每个状态填满：

| 当前状态 | 如果输入 0 去哪？ | 如果输入 1 去哪？ |
| -------- | ----------------- | ----------------- |
| 状态 A   | ?                 | ?                 |
| 状态 B   | ?                 | ?                 |

- **意义**：只要表格没填满，就说明漏想了状态或转移条件。这能 100% 消灭 Verilog 综合出的隐式 Latch（锁存器）。



## 数据捕获

| 步骤             | 思考核心                      | 动作建议                                                     |
| ---------------- | ----------------------------- | ------------------------------------------------------------ |
| **Step 1: 定位** | 哪些状态是真正的数据位？      | 划定状态区间（如 `B1` 到 `B8`）。                            |
| **Step 2: 认门** | 是 LSB First 还是 MSB First？ | **LSB** 就往右挤（`{in, q[7:1]}`）；**MSB** 就往左挤（`{q[6:0], in}`）。 |
| **Step 3: 采样** | 要不要等 Done 信号？          | 数据是在传输过程中**实时存入**的，但只有在 `done` 为 1 时才**对外宣布有效**。 |
| **Step 4: 清空** | 复位时要清零吗？              | 题目说 `done` 以外是 don't-care，意味着你不用刻意清零，但给个初始值是好习惯。 |
